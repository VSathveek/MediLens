00

System overview

RAG stands for Retrieval-Augmented Generation. Instead of asking the language model to answer from its own memory (where it can hallucinate), MediLens first retrieves relevant passages from a curated corpus of biomedical literature, then hands those passages to the model as grounding context. The answer is written by the LLM but anchored to real PubMed text, and the same passages are returned to the user as citations.

The system splits cleanly into two halves:

  • Offline (build once): an ingestion pipeline curates and vectorizes the corpus into three artifacts committed to the repo.
  • Online (per request): a FastAPI service loads those artifacts into memory and serves ranked retrieval with a single sparse matrix multiply — no network call.

01

The corpus & data

The knowledge source is a large dump of PubMed abstract chunks. Only a curated subset is actually served; the raw dump stays local as a research artifact and never ships to production.

Raw source
medilens_rag/texts.json — a JSON array of 262,306 PubMed abstract chunks (~245 MB, gitignored)
Avg chunk size
~1,100 characters of clinical/biomedical prose per chunk
Served subset
20,000 chunks after filtering, deduping, and sampling
Filter window
Keep chunks between 150 and 2,200 characters
Determinism

The build emits exactly three files into backend/data/, and these are the only data the running service needs:

ArtifactWhat it isSize
rag_vectorizer.pklThe fitted TfidfVectorizer — vocabulary + IDF weights. Used to turn a query string into a vector in the same space as the documents.2.6 MB
rag_matrix.npzThe document–term matrix: one pre-computed TF-IDF row per chunk, stored as a SciPy sparse (CSR) matrix.13 MB
rag_corpus.jsonThe chunk texts in the same row order as the matrix, so a matched row index maps back to readable text.19 MB
Why three separate files

The vectorizer transforms queries, the matrix holds the pre-computed document vectors so nothing is re-embedded at request time, and the corpus is the index→text lookup for building context and citations. Splitting them keeps each concern independent and each file small enough to commit directly — no Git LFS, no runtime download.

02

Ingestion pipeline

The whole pipeline lives in scripts/build_rag_corpus.py and runs offline, on a laptop, in seconds. It never runs in production and needs no API key. Run it with python scripts/build_rag_corpus.py --target 20000.

1
Load raw chunks
Read all 262,306 entries from texts.json into memory.
2
Filter by length
Drop chunks shorter than 150 or longer than 2,200 chars — removes fragments and page-scale noise that hurt TF-IDF weighting.
3
Deduplicate
A set membership check drops exact-duplicate chunks so no passage is over-represented.
4
Shuffle & sample
Seeded shuffle (Random(42)), then take the first --target (20,000) — an unbiased random subset that fits in RAM.
5
Fit TF-IDF
TfidfVectorizer.fit_transform(chunks) learns the vocabulary + IDF and produces the 20,000 × 60,000 document–term matrix in one pass.
6
Persist artifacts
Pickle the vectorizer, save_npz the sparse matrix, dump the corpus JSON — all into backend/data/.
Offline ingestion — build once, commit, never re-run at request time
scripts/build_rag_corpus.pypython
# --- steps 2-4: filter, dedupe, seeded sample ---
def select_chunks(target):
    texts = json.load(open(SOURCE_TEXTS, encoding="utf-8"))
    seen, candidates = set(), []
    for t in texts:
        t = t.strip()
        if not (150 <= len(t) <= 2200):   # length filter
            continue
        if t in seen:                        # dedupe
            continue
        seen.add(t); candidates.append(t)
    random.Random(42).shuffle(candidates)      # deterministic
    return candidates[:target]

# --- step 5: fit the index ---
vectorizer = TfidfVectorizer(
    max_features=60000,      # cap vocabulary to top-60k terms
    ngram_range=(1, 2),      # unigrams + bigrams
    stop_words="english",     # drop "the", "of", "and"...
    sublinear_tf=True,         # 1 + log(tf), dampens repeats
    min_df=2,                # term must appear in >= 2 docs
)
matrix = vectorizer.fit_transform(chunks)   # (20000, 60000) sparse

# --- step 6: persist ---
pickle.dump(vectorizer, open(VECTORIZER_OUT, "wb"))
sparse.save_npz(MATRIX_OUT, matrix)
json.dump(chunks, open(CORPUS_OUT, "w"))

Why each vectorizer parameter matters

ParameterValueReasoning
ngram_range(1, 2)Unigrams catch single terms (metformin); bigrams capture medical phrases (blood pressure, heart failure) that carry meaning single words lose.
max_features60,000Keeps only the 60k most frequent terms — bounds matrix width and memory, discards the long tail of rare noise.
min_df2A term must appear in at least 2 documents. Kills one-off typos/OCR garbage that would otherwise become useless vocabulary columns.
sublinear_tfTrueUses 1 + log(tf) instead of raw counts, so a word appearing 10× isn't treated as 10× more important than appearing once.
stop_words"english"Removes function words that appear everywhere and carry no retrieval signal.
03

TF-IDF, at the low level

TF-IDF (Term Frequency – Inverse Document Frequency) scores how important a term is to a specific document relative to the whole corpus. A term gets a high weight when it appears often in this document but rarely across all documents — which is exactly what makes a word distinctive.

The weight of one term in one document

With the settings above, the weight scikit-learn computes is:

Formula

tf = 1 + log(count)  (sublinear)
idf = log((1 + N) / (1 + df)) + 1  (smoothed)
weight = tf × idf  →  row L2-normalized

Where N is the number of documents (20,000) and df is how many of them contain the term. After weighting, each document row is L2-normalized (scaled to unit length). That normalization is the key detail that makes scoring cheap — see below.

Documents become vectors in a shared space

Every document is a point in a 60,000-dimensional space (one dimension per vocabulary term), but almost all coordinates are zero — a typical abstract uses only a few hundred distinct terms. That's why the matrix is stored sparse: it holds only the non-zero entries, turning what would be a 20,000 × 60,000 = 1.2-billion-cell dense array into ~13 MB.

Similarity = cosine = a dot product

Relevance between a query and a document is their cosine similarity — the cosine of the angle between their vectors, ranging 0 (nothing in common) to 1 (identical direction). Cosine is:

Why the code is just a matrix multiply

cosine(a, b) = (a · b) / (‖a‖‖b‖). Because both document and query vectors are already L2-normalized (‖a‖ = ‖b‖ = 1), the denominator is 1, so cosine similarity collapses to a plain dot product. Scoring all 20,000 documents is therefore one sparse matrix × vector product: matrix @ query.T.

04

Retrieval at query time

At runtime the RagStore (in backend/app/rag.py) loads the three artifacts once at process startup and keeps them in memory for the life of the container. Every search reuses them.

backend/app/rag.pypython
def search(self, query, top_k):
    if not self.ready:
        return []
    query_vec = self._vectorizer.transform([query])   # 1) query → TF-IDF vector
    scores = (self._matrix @ query_vec.T).toarray().ravel()  # 2) cosine vs every doc
    if not scores.any():                             # no term overlap at all
        return []
    top_indices = scores.argsort()[::-1][:top_k]     # 3) rank, take top-k
    return [RagHit(text=self._corpus[i], score=float(scores[i]))
            for i in top_indices if scores[i] > 0]  # drop zero-score
1
Vectorize the query
vectorizer.transform([query]) applies the same learned vocabulary + IDF, placing the query in the identical 60k-dim space as the documents. Terms not in the vocabulary are silently ignored.
2
Score against all documents
matrix @ query.T — one sparse multiply produces a similarity score for all 20,000 documents at once. Sub-millisecond.
3
Rank & select top-k
argsort()[::-1][:top_k] sorts descending and takes the best top_k = 5; anything with a zero score (no shared terms) is dropped.
4
Feed the generator
The pipeline keeps the top 3 hits, truncates each to 500 chars, joins them into a context block, and passes it to Gemini as grounding. The same hits become the citations shown to the user.
Query-time retrieval — from raw text to ranked, grounded context
The query is rewritten first

Retrieval doesn't run on the user's raw sentence. Gemini's triage call rewrites "I've had a cough and mild fever for three days" into keyword-dense medical terms like "persistent cough, mild fever, respiratory infection". Lexical retrieval like TF-IDF matches on shared words, so a keyword query overlaps far better with terse abstract text than conversational phrasing does.

05

Full request lifecycle

Putting retrieval back in context — here is everything that happens for one POST /api/process, orchestrated by backend/app/pipeline.py.

1
Triage + rewrite + NER — Gemini call #1
One structured-JSON call returns in_scope, a rewritten_query, and extracted entities (symptoms, medications, conditions, body parts). Temperature 0.1 for consistency; a Pydantic response_schema forces valid JSON.
2
Scope gate
If in_scope is false, return a polite refusal immediately — no retrieval, no second LLM call. This short-circuit saves latency and quota on off-topic questions.
3
Retrieve — local, no API
rag_store.search(rewritten_query, top_k=5), then keep the top 3 as context.
4
Generate answer — Gemini call #2
The question + retrieved excerpts + system prompt go to Gemini (temperature 0.4) to write a patient-friendly, grounded explanation.
5
Assemble response
Return answer, flattened entities, and references (the retrieved passages + their similarity scores) as JSON to the frontend.
Two Gemini calls for an in-scope question · zero for out-of-scope
06

Design decisions & tradeoffs

Why TF-IDF instead of dense vector embeddings?

The original design used Gemini's embedding API with a FAISS index. In practice the free-tier embedding quota was exhausted almost immediately — and critically, embeddings would be needed on every query to vectorize the user's search, so retrieval would break for real users after a handful of requests per day. TF-IDF removes that dependency entirely.

Dense embeddings + FAISSTF-IDF (chosen)
Runtime API calls1 per query (embed the query)0
Rate-limit riskHigh — breaks on free tierNone
Memory footprintLarger dense vectors~13 MB sparse — fits 512 MB
Match typeSemantic (meaning)Lexical (shared terms)

Best forParaphrase / synonym recallTerminology-dense text
The honest tradeoff

TF-IDF is lexical: it can't match "heart attack" to "myocardial infarction" unless the query rewrite bridges them. That's a real limitation. It's mitigated here by the LLM query-rewrite step (which tends to emit canonical medical terms) and is an acceptable price for a retriever that is free, dependency-free, and always available — which a broken embedding retriever is not. The LLM-based rewrite is doing some of the semantic lifting that embeddings would otherwise provide.

Why build the index offline and commit it?

Fitting the vectorizer at container startup would slow every cold start and make deploys non-deterministic. Building offline means the exact same index ships everywhere, startup only has to load (not compute), and the artifacts are small enough to live in git.

Why load into memory once (singleton)?

get_rag_store() is a lazy singleton — the 35 MB is read from disk on first use and reused for every subsequent request. Re-reading per request would add disk I/O and deserialization latency to every call.

07

Complexity & scaling

Build time
Seconds — one fit_transform pass over 20,000 chunks, offline
Query latency
Sub-millisecond for the sparse matmul; total request time is dominated by the two Gemini calls, not retrieval
Query cost
O(nnz) — proportional to non-zero entries, i.e. corpus size × avg terms/doc
Memory
~13 MB matrix + 2.6 MB vectorizer + 19 MB corpus text, all resident
Ranking
O(N log N) argsort over N = 20,000 scores — trivial

This is an exhaustive (brute-force) search — every query is scored against every document. That's perfectly fine up to roughly 100k–1M documents. Beyond that, the linear scan becomes the bottleneck and you'd move to an approximate nearest-neighbor index (FAISS, HNSW) or a purpose-built search engine, and likely shard the corpus. At MediLens's scale, brute force is simpler and fast enough.

Note on cold starts

On Render's free tier the service spins down after inactivity. The first request after idling pays a cold-start cost: the container boots and the RagStore loads its 35 MB into memory before serving. Subsequent requests are warm and fast. This is infrastructure behavior, not a bug in retrieval.

Interview Q&A

The questions an interviewer is most likely to ask about this system, with tight answers. Click to expand.

·
QWhat is RAG and why use it in a medical app?›

RAG = Retrieval-Augmented Generation. Rather than trust the LLM's parametric memory, you retrieve relevant passages from a trusted corpus and give them to the model as context. In a medical setting this matters twice over: it reduces hallucination by grounding answers in real PubMed literature, and it lets you show citations so users can see the evidence. It also means you can update knowledge by changing the corpus, without retraining a model.

QWalk me through what happens from user question to answer.›

One Gemini call triages the question (in-scope?), rewrites it into medical keywords, and extracts entities. If out of scope, we refuse immediately. Otherwise we vectorize the rewritten query with the pre-fitted TF-IDF vectorizer, score it against all documents with a sparse matrix multiply, take the top 5 (keep top 3 for context), and pass those excerpts plus the question to a second Gemini call that writes the grounded answer. We return the answer, entities, and the passages as citations.

QWhy TF-IDF instead of vector embeddings?›

The embedding approach needed an API call on every query to vectorize the user's search, and the free-tier quota was exhausted almost instantly — so retrieval would break for real users. TF-IDF runs fully locally with scikit-learn: zero API calls, no rate limit, ~13 MB sparse index that fits a 512 MB instance. The tradeoff is that TF-IDF is lexical, not semantic — but biomedical text is terminology-dense, and the LLM query-rewrite step supplies canonical terms, which recovers much of the semantic benefit.

QWhat similarity metric do you use, and why is scoring just a matrix multiply?›

Cosine similarity. Cosine is the dot product divided by the product of the vectors' magnitudes. scikit-learn L2-normalizes both the document rows and the transformed query to unit length, so both magnitudes are 1 and the denominator vanishes — cosine reduces to a plain dot product. That's why scoring every document is a single sparse matrix @ query.T operation.

QWhy rewrite the query with the LLM before retrieving?›

Lexical retrieval matches on shared words. A conversational sentence ("I feel dizzy when I stand up") shares few exact terms with a terse abstract. Rewriting it to medical keywords ("orthostatic hypotension, dizziness, postural") dramatically improves term overlap and therefore retrieval quality. The rewrite is effectively doing lightweight query expansion / normalization.

QWhy is the document matrix stored sparse?›

The vocabulary is 60,000 terms but any single abstract uses only a few hundred, so ~99% of every row is zero. A dense 20,000 × 60,000 matrix would be 1.2 billion floats (~9 GB); the sparse CSR representation stores only non-zeros and is ~13 MB. Sparse also makes the matmul cheap because it skips all the zeros.

QWhat do ngram_range=(1,2), min_df=2, and sublinear_tf do?›

ngram (1,2): index both single words and adjacent word pairs, so phrases like "blood pressure" become their own features. min_df=2: ignore any term that appears in fewer than 2 documents — removes typos and OCR noise. sublinear_tf: use 1 + log(count) instead of raw counts so a word appearing many times doesn't dominate linearly.

QIs the index built at runtime? How is it deployed?›

No — it's built offline by scripts/build_rag_corpus.py and the three artifacts (vectorizer, matrix, corpus) are committed to the repo. At runtime the service only loads them, once, into a singleton held in memory. This keeps cold starts fast and deploys deterministic.

QHow does it scale? What breaks at 10M documents?›

Retrieval is currently brute-force: every query is scored against every document, which is fine to roughly 100k–1M docs. Past that the linear scan dominates and you'd switch to an approximate nearest-neighbor index (FAISS/HNSW) or a search engine, shard the corpus, and possibly move the index off-process. Memory also grows with corpus size, so you'd stop holding the full text in RAM and fetch it lazily.

QHow do you stop the model from inventing citations?›

The citations are not generated by the LLM — they are the exact passages the retriever returned, with their similarity scores, assembled in code. The model only writes prose grounded in that context. So the references always correspond to real corpus entries, independent of what the model says.

QWhat happens for an out-of-scope or empty-match query?›

Out of scope: the triage call flags in_scope=false and we return a polite refusal before doing any retrieval or second LLM call — saving latency and quota. No term overlap: all scores are zero, search returns an empty list, and the generator answers from the question alone without grounding (a known limitation).

QHow would you improve retrieval quality next?›

Add a hybrid retriever (BM25 for lexical + a dense model for semantic, fused with reciprocal-rank fusion), then a cross-encoder reranker over the top candidates for precision. Add chunk overlap so context isn't cut mid-idea, grow the corpus beyond the 20k sample, and cache frequent queries. If budget allowed a paid embedding tier, embeddings would restore synonym/paraphrase recall that TF-IDF misses.

QWhy keep only the top 3 and truncate to 500 characters?›

Context budget and signal-to-noise. Retrieving 5 gives a small ranking margin, but stuffing all of them (and their full length) into the prompt adds tokens, cost, and lower-relevance text that can distract the model. Top 3 truncated to 500 chars keeps the grounding tight and the most relevant while staying cheap.
